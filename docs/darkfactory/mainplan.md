# Dark Factory: OpenSpecs v4 (Full Autonomy Architecture)

Этот документ описывает финальную архитектуру Dark Factory. Она объединяет изоляцию контекста, строгие контракты и маршрутизацию задач, закрывая уязвимости подделки логов, гейминга метрик, инъекций и сверх-привилегий.

Главный принцип: **Доверие строится не на LLM-оценщиках, а на независимых источниках фактов, разнесённых по разным песочницам (microVM), и детерминированных механизмах вне LLM.**

---

## SPEC-01: Изоляция контекста и Двойная Песочница
**ID:** DF-SPEC-01

### Requirements:
1. **[REQ-01-A] Изоляция Worker'а:** Worker выполняет работу в изолированном Git Worktree (`Sandbox A`).
2. **[REQ-01-B] Песочница Runner'а (Read-Only):** Test-Runner работает в `Sandbox B`. Код из `Sandbox A` монтируется туда **только для чтения**.
3. **[REQ-01-C] Системные скрипты:** Test-Runner использует *свои* системные скрипты верификации, а не `package.json` внутри проекта.

---

## SPEC-02: Множественность доказательств (Proof Multiplicity)
**ID:** DF-SPEC-02

### Requirements:
1. **[REQ-02-A] Структура Proof'ов:** В `OpenSpec` критерии явно запрашивают тип доказательства: Type A (Behavior `exit_code: 0`), Type B (Visual `pixel_diff`), Type C (Heuristics `SonarQube` + Type A).
2. **[REQ-02-B] Zone-Based Security Gate:** Вне LLM. Трогаешь `/auth` -> принудительно добавляется Type A proof.
3. **[REQ-02-C] Планирование ≠ Авторизация:** Security Gate проверяет план, но не заменяет авторизацию в момент исполнения (см. SPEC-08).

---

## SPEC-03: Внешние Эталоны для UI
**ID:** DF-SPEC-03

### Requirements:
1. **[REQ-03-A] External Baseline:** Для Visual Proofs эталон поступает *снаружи* контура (Figma).
2. **[REQ-03-B] Детерминированный Diff:** Сравнение идёт через Pixelmatch.
3. **[REQ-03-C] Жёсткая привязка:** Привязка "задача <-> эталон" фиксируется детерминированно по пути компонента (как Zone-Gate), а не выбирается агентом.

---

## SPEC-04: Парковка и HIL (Human-In-The-Loop)
**ID:** DF-SPEC-04

### Requirements:
1. **[REQ-04-A] Состояние Pending Approval:** Задачи с Type D (субъективная оценка) паркуются для человека.
2. **[REQ-04-B] Gate на Type D:** Type D присваивается только если Zone-Gate подтвердил отсутствие защищённых зон.
3. **[REQ-04-C] Лимит HIL:** Превышение доли задач в `awaiting_human` сигнализирует о сломанной маршрутизации.

---

## SPEC-05: Песочница уровня microVM
**ID:** DF-SPEC-05
**Цель:** Защита от prompt-injection и исполнения вредоносного кода с доступом к секретам.

### Requirements:
1. **[REQ-05-A] microVM-изоляция:** Обе песочницы (Worker и Runner) исполняются в microVM (Firecracker/Kata/gVisor). Каждая задача — отдельное ядро.
2. **[REQ-05-B] Default-deny сеть:** Worker получает доступ только к allow-list (пакетный реестр), неизменяемому изнутри.
3. **[REQ-05-C] Нет продакшн-секретов:** Ключи не монтируются. Если нужна интеграция — используются scoped-токены (SPEC-08).

---

## SPEC-06: Верификация верификатора (Evaluator Locking + Golden-Set)
**ID:** DF-SPEC-06
**Цель:** Защита от reward hacking (обмана оценщика).

### Requirements:
1. **[REQ-06-A] Evaluator Locking:** Скрипты проверок лежат в Sandbox B. Попытка Worker'а изменить их = `fail`.
2. **[REQ-06-B] Golden-Set:** Новый Evaluator (или скилл) допускается только при 100% прохождении эталонного набора задач вне контура.
3. **[REQ-06-C] File-access instrumentation:** Любое обращение Worker'а к файлам тестов логируется и карается провалом.
4. **[REQ-06-D] Isomorphic Check:** Проверка инвариантности логики на эквивалентных задачах (защита от хардкодинга).

---

## SPEC-07: Meta-Loop (Слойные стоп-условия и Circuit Breaker)
**ID:** DF-SPEC-07
**Цель:** Оркестрация, отлов сбоев и контроль бюджета.

### Requirements:
1. **[REQ-07-A] Слойные стоп-условия:** Лимит попыток (3), токенов, таймаутов, глубины делегирования и детектор отсутствия прогресса (два одинаковых сбоя подряд = эскалация).
2. **[REQ-07-B] Circuit breaker:** Отказ провайдера/модели выше порога переключает на fallback.
3. **[REQ-07-C] Бюджет со скользящим окном:** Лимит трат. При исчерпании — авто-сон контура.
4. **[REQ-07-D] Self-aware termination:** Агент обязан уметь задокументировать неустранимое ограничение и остановиться сам.

---

## SPEC-08: Scope-ограниченные креды (Execution-time authorization)
**ID:** DF-SPEC-08
**Цель:** Защита от сверх-привилегированных агентов.

### Requirements:
1. **[REQ-08-A] Least-privilege креды:** Воркер получает scoped-токен (OAuth) только под конкретную задачу.
2. **[REQ-08-B] Boundary:** Все действия проходят через авторизационный барьер в момент исполнения. `fail-closed` по умолчанию.
3. **[REQ-08-C] Полный аудит:** Запись всех действий агента с его временными правами.

---

## SPEC-09: Транзакционный откат
**ID:** DF-SPEC-09
**Цель:** Гарантировать обратимость мутаций.

### Requirements:
1. **[REQ-09-A] Транзакционная обёртка:** Задача исполняется как транзакция с полным откатом при `fail`.
2. **[REQ-09-B] Перехват высокорисковых команд:** Детерминированный фильтр вне LLM перехватывает `rm -rf`, `deploy` и требует Approval.
3. **[REQ-09-C] Headless-совместимость:** Все предохранители работают без интерактивного ввода.

---

> **Итог:** Ни одна из дыр безопасности Dark Factory не закрывается «умным промптом». Все уязвимости перекрыты на архитектурном уровне: гипервизорами (SPEC-05), locked-файлами (SPEC-06), детерминированными гейтами (SPEC-02/08) и транзакциями (SPEC-09).
