# Dark Factory: OpenSpecs v3 (Multi-Sandbox & Proof Routing)

Этот документ описывает финальную архитектуру Dark Factory. Она объединяет изоляцию контекста, строгие контракты и маршрутизацию задач, закрывая уязвимости подделки логов и гейминга метрик.

Главный принцип: **Доверие строится не на LLM-оценщиках, а на независимых источниках фактов, разнесённых по разным песочницам.**

---

## SPEC-01: Двойная Песочница (Out-of-Band Runner)
**ID:** DF-SPEC-01
**Цель:** Исключить физическую возможность Worker'а подделать результаты тестов или сфабриковать Proof.

### Requirements:
1. **[REQ-01-A] Изоляция Worker'а:** Worker выполняет свою работу в изолированном Git Worktree (`Sandbox A`).
2. **[REQ-01-B] Песочница Runner'а (Read-Only):** Test-Runner работает в `Sandbox B`. Код из `Sandbox A` монтируется туда **только для чтения**.
3. **[REQ-01-C] Системные скрипты:** Test-Runner использует *свои* системные скрипты верификации (внешние по отношению к коду Worker'а), а не доверяет `package.json` внутри проекта.

### Acceptance Criteria:
*   **When** Worker пишет зловредный скрипт `npm run test`, который всегда выводит `PASS`, **Then** Test-Runner игнорирует его, запускает системный бинарник тестов поверх Read-Only кода и возвращает `FAIL`.
*   **When** Worker пытается записать файл-флаг в директорию тестов во время проверки, **Then** он получает `Read-only file system error`.

## SPEC-02: Множественность доказательств (Proof Multiplicity)
**ID:** DF-SPEC-02
**Цель:** Отказ от одного "класса" на задачу. Реальная задача (например, "добавить кнопку оплаты") состоит из UI, логики и рефакторинга одновременно. Spec должен требовать *набор* доказательств.

### Requirements:
1. **[REQ-02-A] Структура Proof'ов:** В `OpenSpec` критерии приёмки должны явно запрашивать тип доказательства:
    *   *Type A (Behavior):* `exit_code: 0` от Runner'а.
    *   *Type B (Visual):* Детерминированный `pixel_diff < 5%` от эталонного Figma-артефакта.
    *   *Type C (Heuristics):* Улучшение метрик (SonarQube) **ПРИ УСЛОВИИ** сохранения поведения (Type A tests == green).
2. **[REQ-02-B] Zone-Based Security Gate:** Ввести `SecurityGate` вне LLM. Если Worker трогает файлы в защищённых директориях (`/auth`, `/payments`), система **принудительно** добавляет требования Type A (строгие тесты) в Spec, даже если Planner забыл это сделать.

### Acceptance Criteria:
*   **When** Worker пытается "оптимизировать" класс (Type C), размазав логику по функциям, **Then** Test-Runner проверяет, что базовые тесты (Type A) остались зелёными. Если нет — провал, метрики игнорируются.
*   **When** Planner размечает задачу в папке `/auth` только как Type C, **Then** `SecurityGate` отклоняет план на старте, требуя добавить `exit_code` тесты безопасности.

## SPEC-03: Внешние Эталоны для UI
**ID:** DF-SPEC-03
**Цель:** Исключить "регресс судей" (когда LLM-Evaluator субъективно оценивает UI, созданный LLM-Worker'ом). 

### Requirements:
1. **[REQ-03-A] External Baseline:** Для задач Type B (Visual) эталон (`baseline.png` или точный CSS-контракт) должен поступать *снаружи* контура (от человека-дизайнера, экспорта из Figma).
2. **[REQ-03-B] Детерминированный Diff:** Сравнение происходит через детерминированные утилиты (Pixelmatch), а не через Vision-LLM. Vision-модели могут использоваться только как советники (помогающие Worker'у понять ошибку), но источником истины является `pixel_diff`.

### Acceptance Criteria:
*   **When** Worker отдаёт кнопку, сдвинутую на 10px, **Then** детерминированный алгоритм вычисляет `diff > 0%`, и задача проваливается без субъективного мнения LLM.

## SPEC-04: Парковка и HIL (Human-In-The-Loop)
**ID:** DF-SPEC-04
**Цель:** Обработка задач, где детерминированная проверка невозможна (вкус, копирайтинг, "чистота" UX), без разрушения автоматического контура.

### Requirements:
1. **[REQ-04-A] Состояние Pending Approval:** Если задача запрашивает Type D Proof (Субъективная оценка) и все остальные пруфы собраны, Оркестратор "паркует" ветку (не делает сбой).
2. **[REQ-04-B] Асинхронный Approve:** Человек получает уведомление со ссылкой на превью. После нажатия Approve ветка вливается, и контур продолжает работу.

### Acceptance Criteria:
*   **When** задача требует только UI-ревью человека, **Then** она не исчерпывает "лимит попыток" Meta-Loop'а, а переходит в статус `awaiting_human`, освобождая ресурсы контура.

> [!IMPORTANT]
> **Выводы по архитектуре**
> Закрыв дыру в категоризации через **Zone-Based Security Gate**, защитив рефакторинг "якорем" поведения (тесты обязаны остаться зелёными), и вынеся Test-Runner в **отдельную Read-Only песочницу**, мы получили систему, которая устойчива к галлюцинациям, попыткам взлома со стороны Worker'а и манипуляциям с метриками.
