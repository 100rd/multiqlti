# Dark Factory: OpenSpecs v6 — Consolidated Architecture

Финальная консолидация. v6 = v5 (девять SPEC'ов, закрывающих подделку логов, гейминг метрик, reward hacking, инъекции, сверх-привилегии — всё механизмами вне LLM) **плюс три слоя, добавленных в последующих раундах:**

1. **Инфраструктурный контур** — система доставляет результат кода в среды (миграции, деплой, зависимости), а не только пишет код. Новый класс задач Type E + SPEC-10.
2. **Сервис памяти со связями и временем** — общий источник истины о состоянии платформы для агентов всех типов; три слоя с разным временем жизни.
3. **Автозапуск планирования** — два режима (A: самозапуск исполнения по правилу; B-narrow: самозапуск ресёрча/планирования, но не действия).

**Сквозной принцип, не сдвинутый ни разу за всю эволюцию:** автономия неоднородна и обрывается ровно на границе побочного эффекта. Думать, исследовать, планировать, прогонять в эфемерной среде — автономно. Необратимое (применение к persistent-состоянию, депрекейт, интерпретация намерения, определение корректности) — за человеком. Это не слабость на пути к L5, это и есть зрелый дизайн: целимся в **уровень 4** (автономное исполнение при человеческом определении корректности), а не в L5.

# ЧАСТЬ I. Ядро v5 (девять SPEC'ов) — кратко

Полные формулировки — в v5. Здесь сжато, для контекста.

- **SPEC-01** Двойная песочница: Worker в Sandbox A, Test-Runner в Sandbox B (read-only mount), системные скрипты верификации, не `package.json` проекта.
- **SPEC-02** Proof Multiplicity: Type A (behavior/exit_code), B (visual/pixel_diff), C (heuristics+A); Zone-Based Security Gate вне LLM; план ≠ авторизация.
- **SPEC-03** Внешние эталоны UI: baseline из Figma, детерминированный Pixelmatch (не Vision-LLM), привязка задача↔эталон по пути компонента.
- **SPEC-04** HIL: Type D паркуется человеку; gate на присвоение Type D; лимит доли `awaiting_human`.
- **SPEC-05** microVM (Firecracker/Kata/gVisor), default-deny сеть, нет продакшн-секретов.
- **SPEC-06** Верификация верификатора: locking + golden-set + file-instrumentation + isomorphic check (классы A/C) + **spec-derived tests** (тесты из спеки, не из реализации — против circularity of error).
- **SPEC-07** Meta-Loop: слойные стоп-условия, circuit breaker, бюджет-окно, self-aware termination, hacker-fixer loop (red-team из другого семейства моделей).
- **SPEC-08** Scope-ограниченные креды: least-privilege OAuth, execution-time authorization (fail-closed), полный аудит.
- **SPEC-09** Транзакционный откат: обёртка с rollback, перехват высокорисковых команд вне LLM, headless-совместимость.

# ЧАСТЬ II. Инфраструктурный контур (новое)

Как только система доставляет результат кода в среду, появляются **действия с побочными эффектами, переживающие git-откат**. Откатить коммит легко; откатить миграцию, дропнувшую колонку, — нет. Центр тяжести смещается с «корректности кода» на «обратимости действия».

## SPEC-E0: Type E — класс инфраструктурных задач
**Цель:** отдельный класс для side-effecting действий (миграции, деплой, изменения зависимостей), у которых своя модель доказательства и обратимости.

**Requirements:**
- **[REQ-E0-A]** Триада доказательств для миграции: (1) forward применилась, (2) **rollback применился** — down-скрипт обязателен и тестируется, его отсутствие = `fail` до исполнения, (3) поведение на prod-подобных данных не сломалось.
- **[REQ-E0-B]** Zone-Gate расширен на инфру: задача, трогающая миграции/деплой/зависимости, **не может** быть Type A или D — принудительно Type E со всеми его proof'ами.
- **[REQ-E0-C]** Deploy-proof: приложение поднимается с зависимостями в prod-подобной среде, проходит поведенческую проверку (smoke/integration), и только тогда verdict.

**Почему exit_code недостаточно.** Миграция зелёная на пустой тестовой базе ложится на проде из-за объёма или грязных данных. Type A проверяет «код отработал», Type E проверяет «действие безопасно применимо и откатываемо на реалистичных данных».

## SPEC-E1: Три среды — три уровня автономии
**Цель:** автономия растёт неравномерно по средам; это сознательный дизайн.

| Среда | Обратимость | Уровень автономии | Механизм |
|-------|-------------|-------------------|----------|
| **Ephemeral (динамические энвы)** | Выбрасыванием среды | Максимальный | Каждый прогон — свежий env из IaC, уничтожается после verdict. «Откат» = «среда не жила дольше теста» |
| **Persistent prod-like** | Re-provision к baseline | Средний | Среда переживает задачу → межзадачная интерференция. Возврат к чистому baseline между задачами детерминированным re-provision, **не агентом** |
| **Prod (read-only)** | Не пишем вообще | Только чтение | Источник данных, не цель действия (см. Часть III) |

**Requirements:**
- **[REQ-E1-A]** Ephemeral по умолчанию: поведенческие тесты (миграции, интеграция) идут в одноразовых средах.
- **[REQ-E1-B]** Persistent prod-like: возврат к baseline между задачами — системный re-provision, не самоочистка агента (агенту нельзя доверять уборку за собой).
- **[REQ-E1-C]** Применение миграции к persistent-среде с непустым состоянием = кандидат на необратимость → approval (SPEC-04). Эфемерная среда — нет (нечего терять). Дифференциация по среде, не по типу действия.

# ЧАСТЬ III. Сервис памяти: observability как источник истины

Prod-наблюдения играют две роли: **proof на выходе** (не деградировало) и **вход в планирование** (требование, выведенное из реальности). Второе мощнее, но опаснее — потому что наблюдение ≠ намерение.

## SPEC-10: Observation/Memory Service
**Цель:** общий источник истины о состоянии платформы для агентов всех типов, со связями и растянутый по времени. Изолирован от исполнения недоверенного кода.

### Три слоя памяти с разным временем жизни

**Слой 1 — Observation Store (наблюдения).**
Сырые агрегаты и тренды из прода. Изменчивый, append-only, retention-политика.
- **[REQ-10-A]** PII-фильтрация на входе: логи с PII недоступны агенту; лимитирование по типу/тегам. Канал отдаёт **агрегаты и метрики, не сырые строки** — агент видит «p99 = 340ms», не содержимое запросов.
- **[REQ-10-B]** Трендовое хранение, не точечное: распределение во времени, чтобы отличать регрессию от суточного колебания. (Вывод «JSON-поле усохло до 5%» существует **только** в ряду во времени, не в снимке.)
- **[REQ-10-C]** Изоляция от исполнения: read-only прод-канал **никогда** не монтируется в Worker-песочницу; отдельный scoped-токен (SPEC-08), полный аудит. Реальные данные не оседают в proof-артефактах и контексте следующей задачи.

**Слой 2 — Derived Requirements (выводы).**
То, что система извлекла из наблюдений: «MTTR-бюджет 1мин», «поле X — кандидат на депрекейт».
- **[REQ-10-D]** Provenance обязателен: каждый вывод хранит ссылку на наблюдение, временное окно и уверенность. Без этого спека гниёт — накапливает требования, чьё обоснование испарилось.
- **[REQ-10-E]** Gate интерпретации: вывод, затрагивающий необратимое (депрекейт API, breaking change), помечается «требует подтверждения человеком». «Трафик упал до 5%» (факт) ≠ «старый формат можно удалить» (решение). Метрика сама не различает легитимную миграцию клиентов от потери одного критичного интегратора.

**Слой 3 — OpenSpec (контракт).**
Стабильный человекочитаемый источник истины.
- **[REQ-10-F]** Спека **ссылается, а не копирует**: «MTTR<1мин (см. derived-req-447)», не хранит сам ряд. Когда наблюдение дрейфует, обновляется вывод, и спека видит, что её основание сдвинулось.
- **[REQ-10-G]** Drift-detector: фоновая задача сравнивает текущее наблюдение с тем, что породило активное требование; флаг при расхождении выше порога.

**Принцип разделения слоёв:** наблюдения изменчивы, выводы интерпретируемы, спека стабильна. Нельзя дать изменчивости течь прямо в стабильное — иначе стабильное перестанет быть источником истины.

### Связи в графе памяти — отдельная осторожность
- **[REQ-10-H]** Рёбра графа (связи между фактами) бывают двух сортов: **наблюдаемые** (трассировка, зависимость в коде) и **гипотезы** (выведенная корреляция). Гипотеза-ребро помечается как требующее подтверждения и не подаётся в планирование как факт. Иначе сервис памяти станет источником уверенных ложных причинностей — хуже, чем отсутствие памяти. («Латентность выросла, потому что вырос JSON» может быть совпадением.)

# ЧАСТЬ IV. Автозапуск планирования: два режима

Самостоятельный старт планирования меняет природу системы — из реактивной (человек ставит цель) в способную **порождать** активность. Развилка решается не «можно/нельзя», а разделением **права инициировать** и **права решать**.

## Режим A — самозапуск исполнения по одобренному правилу (основной)
**Намерение задано человеком заранее.** Триггер автоматический (расписание или дрейф-порог), но «что считать поводом и что с этим делать» — заранее очерчено человеком.
- **[REQ-A]** Drift-detector/cron заводит задачу **только** для обратимого/безопасного класса по заранее одобренному шаблону. Drift-detector здесь — будильник, не стратег.
- Пример: «p99 пробил 500ms → заведи задачу оптимизации по шаблону T в эфемерной среде». Безопасно, потому что это отложенное человеческое решение.

## Режим B-narrow — самозапуск ресёрча и планирования, но НЕ действия
**B на входе, A на выходе.** Система свободно порождает гипотезы, исследует состояние платформы, копает связи в сервисе памяти — но на выходе стоит **вопрос/ранжированные кандидаты человеку, а не задача**.

Почему безопасно: ресёрч и планирование **обратимы по природе** — нулевой blast radius, ничего в мире не изменилось. Круг намерения разорван в правильном месте: думать автономно, действовать — с санкции.

- **[REQ-B-1]** Выход самозапущенного планирования — **вопрос или ранжированное предложение**, не задача. «Поле X усохло до 5% за два квартала, связано с сервисом Y, вот три прочтения — какое верное?», не «я решил депрекейтить X».
- **[REQ-B-2]** Анти-bias агенды: ресёрч предъявляет не только верх списка, но и **что отброшено и почему**. Иначе система влияет на решение человека через подачу (систематически ранжируя «депрекейт» выше «поддержать легаси» — не решает, но подводит). Человек должен остаться судьёй агенды, не только судьёй задач.
- **[REQ-B-3]** Бюджет-gate на ресёрч (из SPEC-07): автономный ресёрч триггерный (дрейф/расписание/накопленный сигнал), не непрерывный. Система, которой нечего делать, не должна исследовать ради исследования.

**Граница между A и B** проходит по линии побочного эффекта: самозапуск, предлагающий обратимое (индекс, оптимизация запроса в эфемерной среде) → дальше в автономию; упирающийся в необратимое или интерпретацию намерения → стоп на вопросе к человеку.

# ЧАСТЬ V. Где остаётся человек (карта, честная для дебатов)

| Что | Автономно | Человек |
|-----|-----------|---------|
| Написание кода | да | — |
| Прогон тестов, verdict по spec-derived проверкам | да | — |
| Исполнение в эфемерной среде | да | — |
| Миграция/деплой в эфемерной среде | да | — |
| Миграция к persistent-состоянию | — | approval |
| Депрекейт / breaking change | — | решение |
| Старт планирования и ресёрча | да | — |
| Назначение цели из ресёрча | — | выбор из кандидатов |
| Интерпретация prod-метрики в требование | — | подтверждение |
| Определение корректности (спека, golden-set) | — | курирование |
| Связь-гипотеза в графе памяти → факт | — | подтверждение |

**Читается одной фразой:** машина обнаруживает, исполняет обратимое и доказывает соответствие; человек интерпретирует, определяет корректность и санкционирует необратимое. Это разделение труда, к которому пришли ещё три исходных доклада — человек поднимается на уровень суждения, а не исчезает.

# ЧАСТЬ VI. Открытые границы (для дебатов, признаём сами)

1. **Reward hacking снижается, не обнуляется** — даже лучшие модели дают ненулевой процент; детектируемость падает без рассуждений в трейсе. Мониторинг хрупок by design.
2. **Корректность спеки и golden-set — на человеке.** Это и есть L4, а не L5. Задачи, прошедшей в прод без человека ни в коде, ни в тесте, ни в эталоне, в архитектуре не существует.
3. **Самопорождение целей (полный B) сознательно не реализовано** — оно пробивает опору «человек определяет корректность» и поднимает circularity с уровня тестов на уровень намерений. B сужен до ресёрча.
4. **Связи в памяти могут быть ложными** — граф причинности, выведенный агентом, галлюцинирует корреляции; защита (REQ-10-H) снижает, не устраняет.
5. **Read-only прод расширяет поверхность утечки** — PII-фильтр и изоляция канала (REQ-10-A/C) обязательны; «autonomy» не означает «доступ ко всему».
6. **Необратимость живёт в данных и среде, не в коде** — ни один LLM-verdict её не страхует, только архитектура сред (эфемерность, re-provision, approval). Автономия неоднородна по средам — это ответ, а не недоделка.

---

## Итог

v6 — это v5 (зрелая защита кодового контура) плюс честное расширение на инфраструктуру, память и автозапуск. Ни одно расширение не сдвинуло границу автономии за линию побочного эффекта. Система обнаруживает проблемы сама, исследует сама, исполняет обратимое сама и доказывает соответствие спеке сама — но интерпретацию, определение корректности и санкцию необратимого держит человек. Это не «фабрика без людей». Это фабрика, где свет выключен на этаже исполнения и горит на этаже суждения — и именно поэтому ей можно доверить продакшн-смежную работу, не дожидаясь, пока модели перестанут галлюцинировать.
